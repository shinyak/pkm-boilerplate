ユーザーからの入力「$ARGUMENTS」を brain に取り込む。

## 1. 入力の取得と判定

- `$ARGUMENTS` が空または極端に短い（20 文字未満）場合は、「何を取り込むか」をユーザーに確認し、本文を受け取ってから進める
- 入力が長文（数百字以上の会議録・議事録・トランスクリプト等）か短文（一言メモ・気づき）かを判定し、後段の出力フォーマットを切り替える

## 2. タイトル slug の生成

- 内容の中心テーマを表す 3〜5 語のスラッグを生成する
- 英数字ハイフン区切り、小文字（例: `issue-prioritization`, `weekly-1on1-memo`）
- 日本語入力でも英語スラッグに変換する

## 3. 関連ノート探索（並列）

Task ツールで Explore エージェントを **並列で** 起動し、以下を検索する。キーワードは入力から 3〜5 個抽出する。

- `00_Inbox/`: 同じトピックの未処理メモ
- `30_Tech_Notes/`: 既存の技術ノート
- `20_Projects/` および `99_Archives/`: 関連するプロジェクト
- `50_Business_Context/`: 関連するドメイン知識

ヒットしたファイルは「パス」と「関連する理由（1 行）」で記録する。

## 4. Inbox ファイルの作成

`00_Inbox/YYYY-MM-DD_<slug>.md` を作成する。

### 短文入力の場合

```markdown
# <タイトル>

<本文そのまま>

---
## 関連ノート候補
- `<path>`: <関連理由>
```

### 長文入力（会議録・議事録等）の場合

```markdown
# <タイトル>

## 要約
- <3〜5 点で要点抽出>

## 決定事項・アクション
- [ ] <あれば。なければセクションごと省略>

## 本文
<原文をそのまま保存>

---
## 関連ノート候補
- `<path>`: <関連理由>
```

関連ノートが 0 件の場合は、`## 関連ノート候補` セクションごと省略する。

## 5. 報告と分類提案

以下を構造化してユーザーに報告する：

1. **作成したファイルパス**
2. **関連ノート**（最大 3 件まで）
3. **分類候補**: この内容の最終的な行き先を 1〜2 個、理由付きで提示
   - 明らかに Inbox 以外が適切（例: 永続的な技術知識 → `30_Tech_Notes/`）なら、即時昇格を提案
   - プロジェクト関連が強いなら、対象プロジェクトへの格納を提案
   - まだ熟成が必要なら Inbox 据え置きを推奨

## 6. ユーザーの指示に応じた後処理

報告後、ユーザーの返答に応じて以下のいずれかを実行する：

- **そのまま Inbox**: 何もしない
- **昇格**: 該当ディレクトリへ移動し、見出し構造とファイル名を最終形に整える
- **統合**: 既存ノートに追記する（マージ方針をユーザーに確認し、元の Inbox ファイルは `99_Archives/` に移すか削除するかも合わせて確認する）

指示が無い場合は Inbox 据え置きを既定とし、何もしない。
